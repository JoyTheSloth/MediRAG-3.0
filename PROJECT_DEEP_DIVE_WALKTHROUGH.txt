MediRAG-Eval 3.0: In-Depth Project Walkthrough & Codebase Guide
=============================================================

This document provides a comprehensive breakdown of the MediRAG-Eval 3.0 
architecture, following the data flow from ingestion to forensic audit.

---

1. PROJECT STRUCTURE
--------------------
The project is a decoupled Full-Stack application:

Backend/ (Python/FastAPI)
- src/api/       : FastAPI routes, Pydantic schemas, and audit logging.
- src/pipeline/  : Core RAG components (chunking, embedding, retrieval).
- src/modules/   : Forensic modules (Faithfulness, NER, Source Credibility).
- src/evaluation : Aggregators and RAGAS integration logic.

Frontend/ (React/Vite)
- src/pages/     : High-fidelity views (MediChat, Dashboard, Patient Experience).
- src/components : Glassmorphic UI elements and custom targeting cursor logic.

---

2. STAGE 1: INGESTION & KNOWLEDGE INFRASTRUCTURE
------------------------------------------------
Before querying, the "Verified Medical Knowledge" must be indexed.

- Chunking (chunker.py): 
  Medical literature is split into semantically dense 512-token chunks. 
  It preserves metadata like 'doc_id', 'title', and 'evidence_tier'.

- Embedding (embedder.py): 
  Each chunk is passed through BioBERT (dmis-lab/biobert-v1.1) to create 
  768-dimension clinical vectors.

- FAISS Indexing: 
  Vectors are stored in a FAISS FlatIP index (Cosine Similarity) for 
  instantaneous semantic retrieval across 100k+ clinical entries.

---

3. STAGE 2: THE HYBRID RETRIEVAL STRATEGY
-----------------------------------------
Located in: src/pipeline/retriever.py

When a user asks a question, MediRAG doesn't just look for "similar concepts." 
It uses a two-pronged approach:

A. Semantic Retrieval (FAISS): Handles conceptual matches (e.g., "Hypertension" 
   matches "High Blood Pressure").
B. Keyword Retrieval (BM25): Handles exact matches for clinical codes, 
   drug names, and specific dosages (e.g., "Metformin 500mg").

The two result sets are fused to provide a high-precision context block.

---

4. STAGE 3: GENERATION & SYSTEM PROMPTING
-----------------------------------------
Located in: src/pipeline/generator.py

MediRAG uses a "Cerebral Grounding" prompt:
- The LLM (Gemini-2 or Mistral-Large) is treated as a clinical auditor.
- It MUST cite sources inline using the format [Source: <title>].
- It is instructed to prioritize the retrieved context over its training data.

---

5. STAGE 4: THE FORENSIC AUDIT ENGINE (CORE LOGIC)
--------------------------------------------------
Located in: src/evaluate.py

Every generated answer is paused before delivery. It passes through:

A. Faithfulness Scorer (faithfulness.py): 
   Uses a DeBERTa-v3 NLI model to check if each claim in the answer is 
   explicitly supported by the retrieved chunks.

B. Entity Verifier (entity_verifier.py): 
   SciSpaCy (en_core_sci_lg) extracts every medical entity. If the LLM 
   specifies 100mg but the source says 10mg, the entity verifier catches 
   the lethal hallucination.

C. Contradiction Detection (contradiction.py): 
   Ensures the LLM doesn't verify and then contradict itself in the 
   same response.

D. Source Credibility Ranking (source_credibility.py): 
   Assigns weight based on the quality of the source (e.g., Randomized 
   Controlled Trials > General Reviews).

---

6. STAGE 5: THE INTERVENTION LOOP & SAFETY GATE
-----------------------------------------------
Located in: src/api/main.py (@app.post("/query"))

Based on the audit, a Health Risk Score (HRS) is generated (0-100).

- GREEN (HRS < 40): Response delivered normally with evidence cards.

- YELLOW (HRS 40 - 85): The "Intervention Gate" triggers REGENERATION. 
  The LLM is called again but in "Strict Strict Mode." It is forbidden 
  from using any outside knowledge. The user sees a "Regenerated for 
  Safety" badge.

- RED (HRS > 86): The "Safety Gate" BLOCKS the response. The user sees 
  a CRITICAL safety warning, preventing the spread of AI hallucinations.

---

7. STAGE 6: GOVERNANCE & DASHBOARDING
-------------------------------------
Every query, answer, audit score, and intervention is logged into 
data/logs.db (SQLite). This allows clinicians to audit why an AI was 
blocked or how it arrived at a specific conclusion, fulfilling medical 
transparency requirements.
